# Automation Tooling Guidelines

This document defines **best practices and restrictions** when working with the internal automation tools.

Tools covered:

- page-scanner
- page-object-generator
- page-object-validator
- page-object-repair
- data-builder

Following these guidelines ensures the automation framework remains **stable, deterministic, and maintainable**.

---

# General Principles

The automation framework follows three core principles:

1. **Generated artifacts must remain deterministic**
2. **Manual code must not be overwritten**
3. **Tools must operate in a predictable pipeline**

Developers should avoid making manual changes that conflict with the toolchain.

---

# Source of Truth Rules

The framework has strict ownership rules.

| Component | Source of Truth |
|----------|----------------|
page elements | `elements.ts`
generated aliases | `aliases.generated.ts`
business aliases | `aliases.ts`
page objects | `PageObject.ts`
metadata | page maps
registry | generator
manifest | generator / repair
test data | data-builder

Important:

**Page maps are metadata only and are NOT the source of truth for elements.**

---

# Page Scanner

## DO

✔ Use the scanner to discover page structure  
✔ Scan pages when UI changes occur  
✔ **Review generated page-map element keys before generating page objects**  
✔ Rename element keys in the page map if they do not match business terminology  
✔ Use meaningful, stable names that represent the business function of the element  

Example:

```
loginButton
submitClaimButton
emailTextbox
driverDOBField
```

Renaming keys **before running the generator** ensures the page-object API remains clean and readable.

Example workflow:

```
1. Run scanner
2. Review page-map keys
3. Rename keys if needed
4. Run generator
```

## DON'T

✘ Do not generate page objects without reviewing element keys  
✘ Do not keep auto-generated generic names if they do not represent business meaning  
✘ Do not treat page maps as the source of truth for elements    

The generator manages:

```
elements.ts
aliases.generated.ts
aliases.ts
PageObject.ts
index.ts
pageManager.ts
.manifest
```

---

# Page Object Validator

## DO

✔ Run validator before committing changes  
✔ Fix warnings and errors immediately  
✔ Use strict mode when validating CI pipelines  

Example:

```
npm run validator:check
```

## DON'T

✘ Do not ignore validator errors  
✘ Do not bypass validator in CI pipelines  

The validator ensures:

- page-chain consistency
- manifest correctness
- registry correctness
- naming conventions

---

# Page Object Repair

## DO

✔ Use repair when validator detects inconsistencies  
✔ Re-run validator after repair  

Example:

```
npm run repair:run
npm run validator:check
```

## DON'T

✘ Do not rely on repair as a normal workflow  
✘ Do not manually edit manifest or registry files  

Repair is intended for **framework recovery only**.

---

# Data Builder

The Data Builder converts **Excel input data** into **JSON test cases**.

Pipeline:

```
Excel
 → LoadExcel
 → ExtractMeta
 → BuildCases
 → FilterScriptIds
 → PruneDrivers
 → WriteJSON
```

Output:

```
src/data/generated/*.json
```

## DO

✔ Keep Excel structure consistent  
✔ Use plugins to extend transformation logic  
✔ Validate generated JSON before running tests  

## DON'T

✘ Do not manually edit generated JSON files  
✘ Do not change plugin execution order without understanding dependencies

# Generated Files Policy

The following files are generated by tools and should **not be edited manually unless explicitly allowed**.

| File | Managed By |
|----|----|
elements.ts | generator
aliases.generated.ts | generator
index.ts | generator
pageManager.ts | generator
.manifest | generator / repair
generated JSON test cases | data-builder

Allowed manual modifications:

- `aliases.ts`
- custom logic inside `PageObject.ts`

# Recommended Developer Workflow

Typical automation workflow:

```
1. Scan page
2. Generate page objects
3. Validate framework
4. Repair if necessary
5. Build test data
6. Run tests
```

Example commands:

```
npm run scan:page
npm run generator:elements
npm run validator:check
npm run repair:run
npm run data:build
```

---

# CI Best Practices

CI pipelines should enforce:

```
generator
validator (strict)
data-builder
tests
```

Example validation step:

```
npm run validator:check:strict
```

# Key Takeaways

✔ Keep the page-object chain intact  
✔ Do not modify generated artifacts  
✔ Use tools instead of manual edits  
✔ Run validator regularly  
✔ Use repair only for framework recovery  

These rules ensure the automation framework remains **stable and scalable**.
